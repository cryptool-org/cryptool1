Hybrid encryption (Menu Crypt/Decrypt \ Hybrid)

The dialog Visualising hybrid encryption is initialised with the active document (if any).

The goal of the hybrid method is to compensate the respective disadvantages of symmetric and asymmetric encryption and to combine their advantages.

The main problem of all symmetric encryption algorithms is the key management for a large number of secret keys (one for each pair of communicating parties). Asymmetric cryptography solves this problem. Symmetric algorithms on the other hand are much faster than the asymmetric ones.

Asymmetric encryption systems are, due to their large computational requirements, used in practice only for encrypting so called "session keys". The pay load is encrypted with a symmetric algorithm using the session key. This combination of two methods is called hybrid encryption (asymmetric/symmetric). This implies no security deficit if you use a good random number generator for the session key.

In hybrid encryption systems the sender "A" creates in a secure environment a "random", i.e. not predictable, symmetric key "S" (the so called "session key") and encrypts the message using this key. The sender then encrypts the session key with the (asymmetric) public key of the recipient "B". Then both, the message encrypted with "S" and the encrypted session key are transmitted to the recipient.

Recipient "B" not knowing the session key, has to decrypt the encrypted session key before he can decrypt the message. He decrypts "S" with his (asymmetric) private key. Now he knows the session key and can use it to decrypt the message that was sent to him.

PGP e.g is based on the hybrid encryption scheme. It uses IDEA for encrypting the data symmetrically and RSA to transport the session key securely.
