Digital signing of messages

  1. Simple, naive procedure: you could encrypt a plaintext using secret key (N,d). The situation has now fundamentally altered:

    2. Anyone can decrypt the encrypted message using public key (N,e). But only you, as owner of the secret key (N,d), could encrypt the message.

    The owner of the secret key encrypts message M using the secret key C(N,d;M). The encrypted version C(N,d;M) can also be viewed as the signature for message M, because the authenticity of the signature can be verified by anyone in possession of C(N,d;M), public key (N,e) and message M:

    The signature C(N,d;M) is valid if the signature that was decrypted with (N,e), M‚ = E(N,e;C), matches the message M.

  2. In practice usually not the message itself is signed, but it is condensed into a hash value of fixed length using a hash function Hand then the hash value signed. This eliminates the following disadvantages of the pure signature C(N,d;M):
    • If a message M is encrypted with the secret RSA key, the amount of time needed for long messages M is very high.
    • The signature C(N,d;M) is at least as long as the signed message M itself.
    • A possible privacy problem, because the message can be recovered from the signature.

    Therefore it is more efficient to sign only the hash value H = H(M) of the message. To verify the signature C = C(N,d;H), one now requires knowledge of the hash function H that has been used as well:

    The signature C(N,d;H) is valid if the signature that was decrypted with (N,e), H‚ = E(N,e;C), matches the message M‚s hash value H(M).
