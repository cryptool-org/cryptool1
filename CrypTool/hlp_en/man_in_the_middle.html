Man-in-the-Middle Attack

By means of a man-in-the-middle attack an attacker tries to jump in, eavesdrop the communication of two parties or to affect untrue identities (he pretends to be the other party towards each party) to gain access.

You can see this within the Challenge-Response demonstration (Network Authentication). There the attacker pretends to be server 1 towards the client and pretends to be the client towards server 1. This is only possible, by using a connection configuration that all data passes the attacker and no other connection is established between server 1 and client.

Operations:

  • The attacker establishes a connection, so all data sent between server 1 and client passes him.

  • Now, the client tries to authenticate and sends a request to server 1. This request is intercepted by the attacker and he sends an own request for authentication to server 1.

  • Therefore server 1 sends a challenge to the attacker.

  • At present the attacker pretends to be server 1 towards the client and sends the challenge to the client to get the correct response.

  • The client believes in communicating with server 1 and returns the response to the attacker.

  • By now the attacker pretends to be the client towards server 1 and sends the received response to server 1.

  • Server 1 gets the expected response from the attacker (who pretends to be the client) and finds acceptance by server 1 as client.

  • The man-in-the-middle attack was successful!

An example for a man-in-the-middle attack is an IMSI-Catcher, which can be used in GSM networks to eavesdrop communication.
