Man-in-the-middle-Angriff

Bei einem „Man-in-the-middle-Angriff“ versucht ein Angreifer, sich in die Kommunikation von zwei Parteien einzuklinken, mitzuhören oder durch Vortäuschen falscher Identitäten (er gibt sich gegenüber den beiden Parteien jeweils als die andere Partei aus = Maskierung) unerlaubten Zugang zu erhalten.

Dies können Sie z.B. in der Challenge-Response-Demo über die Authentisierungsverfahren im Netz sehen. Dort will sich der Angreifer gegenüber Server 1 als Client und gegenüber dem Client als Server 1 ausgeben. Das funktioniert aber nur in einer Verbindungskonfiguration, in der alle Pakete zwischen Server 1 und Client nur über den Angreifer laufen, Server 1 und Client also keine Verbindung mehr zueinander besitzen.

Ablauf:

  • Der Angreifer stellt eine Verbindungskonfiguration her, in der ihn alle Pakete zwischen Server 1 und Client passieren und in der es keine weitere direkte Verbindung zwischen Client und Server 1 gibt.

  • Jetzt versucht sich der Client zu authentifizieren und schickt seinen Wunsch an Server 1. Diese Anforderung fängt der Angreifer ab und stellt eine eigene Anforderung an Server 1, mit dem Wunsch, authentifiziert zu werden.

  • Dafür schickt Server 1 eine Challenge an den Angreifer.

  • Jetzt gibt sich der Angreifer gegenüber dem Client als Server 1 aus und schickt dem Client diese Challenge, um die „richtige“ Response zu erhalten.

  • Der Client denkt, er kommuniziere mit Server 1, und schickt seine Response an den Angreifer.

  • Jetzt gibt sich der Angreifer gegenüber Server 1 als Client aus und schickt die erhaltene Response an den Server 1.

  • Server 1 sieht, dass er die richtige Response vom Angreifer (der sich als Client ausgibt) erhalten hat und authentifiziert den Angreifer als Client.

  • Der Man-in-the-middle-Angriff war erfolgreich!

Ein Beispiel für einen Man-in-the-middle-Angriff ist der IMSI-Catcher, der in GSM-Netzen verwendet werden kann (siehe den Artikel http://www.heise.de/newsticker/meldung/40089).
